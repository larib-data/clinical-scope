description: Generate a database_options.json or .xlsx for a patient data folder by inspecting available datasources and signals, then writing a ready-to-refine config file
---

The user wants to bootstrap a `database_options` configuration file from a patient data folder.

## Before doing anything

Ask the user for:
1. **Patient folder path** (required)
2. **Output format**: JSON or XLSX (default: JSON)
3. **Output path** (default: `<patient_folder>/database_options.json` or `.xlsx`)
4. **Existing database_options to filter by?** (optional — restricts inspection to known datasources only)

If all answers are clear from context, proceed without asking.

## Step 1 — Discover datasources and signals

Run the existing inspect script to get the full signal list:

```bash
python scripts/inspect_patient_data.py <patient_folder> \
    [--database-options <existing_db_options_path>] \
    --output-csv /tmp/cdv_inspection.csv \
    --verbose
```

Read `/tmp/cdv_inspection.csv`. Relevant columns: `datasource`, `status`, `raw_name`, `raw_point_count`.

Skip any datasource row where `status != "ok"` or `raw_name` is empty.

## Step 2 — Auto-parse signal names for label + unit

For each `raw_name`, apply:
- Pattern `"Name(unit)"` → `label = "Name"`, `unit = "unit"`
  - Example: `"Paw(cmH2O)"` → `label: "Paw"`, `unit: "cmH2O"`
  - Example: `"RR(bpm)"` → `label: "RR"`, `unit: "bpm"`
- If no match, or name has multiple parenthetical groups → leave `{}`, user will fill manually
- Never write `label` if it would equal `raw_name` (redundant)

## Step 3 — Build the database_options structure

Follow the `template_database_options.json` structure exactly. Use `_doc` keys for inline hints (user removes them when done).

```json
{
    "_doc": "Auto-generated by generate-database-options skill — remove _doc keys before use.",
    "global": {
        "_doc_grouped_fields": "Group signals from different datasources onto one subplot. Syntax: {\"Group name\": [\"signal_A\", \"signal_B\"]}. Use 'datasource::signal_name' to disambiguate.",
        "grouped_fields": {}
    },
    "<datasource_name>": {
        "_doc": "TODO: review labels, units, range, grouped_fields, loop.",
        "field_display": ["<all raw_names for this datasource>"],
        "signals": {
            "<raw_name>": {"label": "...", "unit": "..."}
        },
        "numerics": {},
        "grouped_fields": {},
        "loop": {},
        "additional_informations": {}
    }
}
```

Datasources appear in the order returned by the inspection CSV (which matches the default plot order from `datasource_list.py`).

## Step 4 — Ask about high-impact refinements

Before writing the file, ask the user if they want to configure any of these (skip if they said "minimal" or "I'll edit manually"):

| Option | Question to ask |
|--------|-----------------|
| `grouped_fields` | Which signals should share a subplot? (e.g., all pressure signals together) |
| `loop` | Any PV-loops? (ventilator: Paw vs Vt, esophageal: Pes vs Vt) |
| `range` | Y-axis limits for any signal? (e.g., `PEEP: [-2, 20]`) |
| `numerics.period_resampling` | Any datasource with too many points / slow loading? (set `0.2` for 20% sampling) |

Incorporate answers into the dict before writing.

## Step 5 — Write the output file

**JSON** (default):
```python
import json
from pathlib import Path
Path(output_path).write_text(json.dumps(db_options, indent=4, ensure_ascii=False), encoding="utf-8")
```

**XLSX**: Use `openpyxl` (already in project dependencies). Write two sheets:

Sheet **"signals"** — header row (bold, light-grey fill `"D9D9D9"`):
```
datasource | signal | label | unit | unit_conversion | color | priority | period_resampling | range_min | range_max | visible | line_dash | hover_template | display | groups | timezone
```
- One sentinel row per datasource (`signal = *`) only if `numerics` has values or `timezone` is set
- One data row per signal (leave cells empty for unset options)
- `timezone` column is **sentinel-only** (placed in `additional_informations.timezone`); a warning is logged if set in a per-signal row

Sheet **"loops"** — header row (bold, light-grey fill):
```
datasource | loop_name | x_signal | y_signal
```
- One row per loop definition

## All configurable options — reference

Use this table to guide refinements and answer user questions.

| Option | Scope | Default | Impact | Notes |
|--------|-------|---------|--------|-------|
| `label` | per-signal | raw_name | 🔴 High | Auto-parsed from `"Name(unit)"` parens |
| `unit` | per-signal | `"-"` | 🔴 High | Auto-parsed from parens |
| `grouped_fields` | per-datasource or global | `{}` | 🔴 High | Signals sharing a subplot |
| `loop` | per-datasource | `{}` | 🔴 High | `{"PV": ["Vt(ml)", "Paw(cmH2O)"]}` |
| `range` | per-signal | auto | 🟡 Med | `[-5, 40]` prevents auto-scaling surprises |
| `priority` | per-signal / numerics | 10000 | 🟡 Med | Lower number = plotted first/higher |
| `color` | per-signal | auto | 🟡 Med | `"red"` or `"#FF5500"` |
| `numerics.period_resampling` | per-datasource | 1.0 | 🟡 Med | `0.2` = load only 20% of points |
| `visible` | per-signal | `true` | 🟢 Low | `false` hides trace from legend by default |
| `line_dash` | per-signal | `"solid"` | 🟢 Low | `"solid"`, `"dash"`, `"dot"`, `"dashdot"` |
| `hover_template` | per-signal | auto | 🟢 Low | `"fraction"` → `1/3`; `"percentage"` → `33.3%`; any other string = raw Plotly hovertemplate |
| `unit_conversion` | per-signal | 1.0 | 🟢 Low | Multiplier applied to raw values |
| `period_resampling` | per-signal | datasource default | 🟢 Low | Fine-grained override per signal |
| `additional_informations.timezone` | per-datasource | source default | 🟢 Low | e.g., `"Europe/Paris"` — only some sources support it |

### Known limitations (tell the user if relevant)

- ⚠️ **`global.loop`** (cross-datasource PV-loops): **not implemented** — `wrapper.py:222-224` logs an error and skips. Only per-datasource `loop` entries work.
- ℹ️ **Qualified signal references** (`"datasource::signal_name"`): supported in `grouped_fields` to resolve ambiguity when the same signal name exists in multiple datasources. Useful in `global.grouped_fields`.
- ℹ️ **`other` datasource — per-file config**: the `other` datasource uses `other::<stem>` keys (e.g. `other::waves`, `other::numerics`) in `database_options` to configure each file independently. In the XLSX, use `other::waves` as the `datasource` value for signals from the `waves.*` file. The `timezone` sentinel column works correctly for these keys.
- ⚠️ **`other` datasource — global groups with `other` signals**: in `global.grouped_fields`, signals from `other` files must be referenced as `"other::waves::art"` (i.e. `datasource::stem::column`). The XLSX `groups` column only generates bare signal names in global groups, so cross-datasource groups involving `other` signals need manual adjustment in JSON after export.
