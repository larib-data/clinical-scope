The user wants to bootstrap a `database_options` configuration file from a patient data folder.

## Before doing anything

Ask the user for:
1. **Patient folder path** (required)
2. **Output format**: JSON or XLSX (default: JSON)
3. **Output path** (default: `<patient_folder>/database_options.json` or `.xlsx`)
4. **Existing database_options to filter by?** (optional вЂ” restricts inspection to known datasources only)

If all answers are clear from context, proceed without asking.

## Step 1 вЂ” Discover datasources and signals

Run the existing inspect script to get the full signal list:

```bash
source /Users/alexis/Codes/clinical_visu_venv/bin/activate
python scripts/inspect_patient_data.py <patient_folder> \
    [--database-options <existing_db_options_path>] \
    --output-csv /tmp/clinical_scope_inspection.csv \
    --verbose
```

Read `/tmp/clinical_scope_inspection.csv`. Relevant columns: `datasource`, `status`, `raw_name`, `raw_point_count`.

Skip any datasource row where `status != "ok"` or `raw_name` is empty.

## Step 2 вЂ” Auto-parse signal names for label + unit

For each `raw_name`, apply:
- Pattern `"Name(unit)"` в†’ `label = "Name"`, `unit = "unit"`
  - Example: `"Paw(cmH2O)"` в†’ `label: "Paw"`, `unit: "cmH2O"`
  - Example: `"RR(bpm)"` в†’ `label: "RR"`, `unit: "bpm"`
- If no match, or name has multiple parenthetical groups в†’ leave `{}`, user will fill manually
- Never write `label` if it would equal `raw_name` (redundant)

## Step 3 вЂ” Build the database_options structure

Follow the `template_database_options.json` structure exactly. Use `_doc` keys for inline hints (user removes them when done).

```json
{
    "_doc": "Auto-generated by generate-database-options skill вЂ” remove _doc keys before use.",
    "global": {
        "_doc_grouped_fields": "Group signals from different datasources onto one subplot. Syntax: {\"Group name\": [\"signal_A\", \"signal_B\"]}. Use 'datasource::signal_name' to disambiguate.",
        "grouped_fields": {}
    },
    "<datasource_name>": {
        "_doc": "TODO: review labels, units, range, grouped_fields, loop.",
        "field_display": ["<all raw_names for this datasource>"],
        "signals": {
            "<raw_name>": {"label": "...", "unit": "..."}
        },
        "numerics": {},
        "grouped_fields": {},
        "loop": {},
        "additional_informations": {}
    }
}
```

Datasources appear in the order returned by the inspection CSV (which matches the default plot order from `datasource_list.py`).

## Step 4 вЂ” Ask about high-impact refinements

Before writing the file, ask the user if they want to configure any of these (skip if they said "minimal" or "I'll edit manually"):

| Option | Question to ask |
|--------|-----------------|
| `grouped_fields` | Which signals should share a subplot? (e.g., all pressure signals together) |
| `loop` | Any PV-loops? (ventilator: Paw vs Vt, esophageal: Pes vs Vt) |
| `range` | Y-axis limits for any signal? (e.g., `PEEP: [-2, 20]`) |
| `numerics.period_resampling` | Any datasource with too many points / slow loading? (set `0.2` for 20% sampling) |

Incorporate answers into the dict before writing.

## Step 5 вЂ” Write the output file

**JSON** (default):
```python
import json
from pathlib import Path
Path(output_path).write_text(json.dumps(db_options, indent=4, ensure_ascii=False), encoding="utf-8")
```

**XLSX**: Use `openpyxl` (already in project dependencies). Write two sheets:

Sheet **"signals"** вЂ” header row (bold, light-grey fill `"D9D9D9"`):
```
datasource | signal | label | unit | unit_conversion | color | priority | period_resampling | range_min | range_max | visible | line_dash | hover_template | display | groups | timezone
```
- One sentinel row per datasource (`signal = *`) only if `numerics` has values or `timezone` is set
- One data row per signal (leave cells empty for unset options)
- `timezone` column is **sentinel-only** (placed in `additional_informations.timezone`); a warning is logged if set in a per-signal row

Sheet **"loops"** вЂ” header row (bold, light-grey fill):
```
datasource | loop_name | x_signal | y_signal
```
- One row per loop definition

### Known limitations (tell the user if relevant)

- в„№пёЏ **`global.loop`** (cross-datasource phase loops): **supported** вЂ” see
  `wrapper.py::_resolve_signal_references`. Two signals per loop, each resolved via the
  3-mode chain (qualified `datasource::raw_name` в†’ display name в†’ raw name fallback).
  See `docs/user_guide/tutorial.md` в†’ *Global Loops vs. Per-Source Loops*.
- в„№пёЏ **Qualified signal references** (`"datasource::signal_name"`): supported in both
  `grouped_fields` and `loop` (global and per-source) to resolve ambiguity when the same
  signal name exists in multiple datasources. Use them in `global.grouped_fields` and
  `global.loop`.
- в„№пёЏ **`other` datasource вЂ” per-file config**: the `other` datasource uses `other::<stem>`
  keys (e.g. `other::waves`, `other::numerics`) in `database_options` to configure each file
  independently. In the XLSX, use `other::waves` as the `datasource` value for signals from
  the `waves.*` file. The `timezone` sentinel column works correctly for these keys.
- вљ пёЏ **`other` datasource вЂ” global groups/loops with `other` signals**: in
  `global.grouped_fields` and `global.loop`, signals from `other` files must be referenced as
  `"other::waves::art"` (i.e. `datasource::stem::column`). The XLSX `groups` column only
  generates bare signal names in global groups, so cross-datasource groups involving `other`
  signals need manual adjustment in JSON after export.
